# Supply-chain attestations

PliegoRS keeps release bytes and supply-chain attestations in two related exact
sets. The existing Ed25519 bundle remains the continuity authority for release
assets. P8 adds an independently generated attestation package without changing
or re-signing that historical contract.

## Attestation package

Every candidate that reaches the golden path produces:

- `PLIEGORS.cdx.json`: a normalized CycloneDX SBOM generated by pinned
  `cargo-cyclonedx 0.5.9` from the exact source revision;
- `PLIEGORS.intoto.jsonl`: an in-toto Statement v1 using the SLSA provenance v1
  predicate, with every sealed release file and the SBOM as digest-bound
  subjects;
- `ATTESTATIONS.json`: an exact manifest binding the release manifest, SBOM,
  provenance, and offline verifier files;
- `ATTESTATIONS.sigstore.json`: a keyless Sigstore bundle over
  `ATTESTATIONS.json`; and
- the verifier and its bounded support library.

The workflow identity is the exact `celiumsai/pliegors` release workflow and Git
ref. Signing uses GitHub OIDC, an ephemeral key, Fulcio, and the Sigstore
transparency service. The project-controlled Ed25519 release key remains a
separate continuity mechanism.

## Verification

After downloading the sealed release into `release-assets` and attestations
into `attestations`:

```sh
node attestations/verify-supply-chain-attestations.mjs \
  --release release-assets \
  --attestations attestations

cosign verify-blob \
  --bundle attestations/ATTESTATIONS.sigstore.json \
  --certificate-identity \
    'https://github.com/celiumsai/pliegors/.github/workflows/release.yml@refs/heads/main' \
  --certificate-oidc-issuer https://token.actions.githubusercontent.com \
  attestations/ATTESTATIONS.json
```

The local verifier rejects missing or extra package files, release-manifest
substitution, SBOM/provenance changes, source revision drift, and any changed
release subject. Cosign separately verifies identity, certificate, signature,
and transparency evidence.

## Claim boundary

The provenance document is **SLSA v1-compatible**. PliegoRS does not claim a
SLSA build level merely because a statement exists. A level claim requires the
complete builder, isolation, distribution, and independent verification
requirements to be demonstrated against an actual hosted candidate.

Local tests prove schema, exact-set, and tamper behavior. They do not prove that
GitHub OIDC, Fulcio, Rekor, or the hosted runner executed successfully. That
evidence is recorded only after a candidate workflow run completes.
